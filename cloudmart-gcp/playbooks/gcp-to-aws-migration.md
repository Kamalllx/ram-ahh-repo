# GCP → AWS Migration Playbook

This document describes the migration of CloudMart from Google Cloud Platform to Amazon Web Services using the **MigrAIne** migration tool.

## Overview

| Phase | Description | Duration |
|-------|-------------|---------|
| 1 | Assessment & Planning | Day 1 |
| 2 | AWS Infrastructure Provisioning | Day 2-3 |
| 3 | Data Migration | Day 3-4 |
| 4 | Application Deployment | Day 4-5 |
| 5 | Traffic Cutover | Day 5-6 |
| 6 | Validation & Cleanup | Day 6-7 |

---

## Service Mapping

### Compute Plane

| GCP Service | GCP Resource | AWS Equivalent | AWS Resource |
|-------------|-------------|----------------|-------------|
| GKE | `cloudmart-cluster` | EKS | `cloudmart-eks-cluster` |
| Cloud Run (API GW) | `cloudmart-api-gateway` | ECS Fargate | `cloudmart-api-gw-task` |
| Cloud Run (User Svc) | `cloudmart-user-service` | ECS Fargate | `cloudmart-user-svc-task` |
| Cloud Functions | 3 functions | Lambda | 3 Lambda functions |
| Compute Engine | `cloudmart-inventory-worker` | EC2 | `cloudmart-worker` (t3.medium) |

### Control Plane

| GCP Service | AWS Equivalent |
|-------------|---------------|
| VPC + Subnets | VPC + Subnets |
| Cloud Load Balancing | Application Load Balancer (ALB) |
| Cloud Armor | AWS WAF + Shield Standard |
| Cloud DNS | Route 53 |
| IAM + Service Accounts | IAM Roles + Instance Profiles |
| Secret Manager | AWS Secrets Manager |
| Cloud KMS | AWS KMS |
| Artifact Registry | ECR |
| Cloud Build | CodePipeline + CodeBuild |

### Data Plane

| GCP Service | AWS Equivalent | Migration Method |
|-------------|---------------|-----------------|
| Cloud SQL (PostgreSQL) | RDS PostgreSQL | AWS DMS |
| Firestore | DynamoDB | Custom migration script |
| Cloud Storage | S3 | `gsutil` → `aws s3 sync` |
| Pub/Sub | SNS + SQS | Re-deploy publishers/consumers |
| Memorystore (Redis) | ElastiCache (Redis) | Redis DUMP/RESTORE |
| BigQuery | Redshift / Athena | Parquet export → S3 import |

## Phase 1: Assessment

### Run MigrAIne Analysis

```bash
cd MigrAI
python main.py analyze \
  --source gcp \
  --project cloudmart-gcp \
  --output assessment-report.json
```

MigrAIne will:
1. Enumerate all GCP resources via API
2. Map each resource to its AWS equivalent
3. Identify dependencies and migration order
4. Flag potential blockers (e.g., GCP-specific APIs)
5. Generate a Terraform plan for AWS infrastructure

## Phase 3: Data Migration

### 3a. PostgreSQL (Cloud SQL → RDS)

Using AWS Database Migration Service (DMS):

```bash
# 1. Create RDS instance (done by Terraform)
# 2. Create DMS replication instance
# 3. Create source endpoint (Cloud SQL via public IP)
aws dms create-endpoint \
  --endpoint-identifier cloudmart-gcp-source \
  --endpoint-type source \
  --engine-name postgres \
  --server-name <CLOUD_SQL_PUBLIC_IP> \
  --port 5432 \
  --database-name cloudmart \
  --username cloudmart_admin \
  --password <DB_PASS>

# 4. Create target endpoint (RDS)
aws dms create-endpoint \
  --endpoint-identifier cloudmart-rds-target \
  --endpoint-type target \
  --engine-name postgres \
  --server-name <RDS_ENDPOINT> \
  --port 5432 \
  --database-name cloudmart \
  --username cloudmart_admin \
  --password <RDS_PASS>

# 5. Run full-load + CDC task
aws dms create-replication-task \
  --replication-task-identifier cloudmart-migration \
  --source-endpoint-arn <SOURCE_ARN> \
  --target-endpoint-arn <TARGET_ARN> \
  --replication-instance-arn <INSTANCE_ARN> \
  --migration-type full-load-and-cdc \
  --table-mappings '{"rules":[{"rule-type":"selection","rule-id":"1","rule-name":"1","object-locator":{"schema-name":"%","table-name":"%"},"rule-action":"include"}]}'
```

### 3b. Firestore → DynamoDB

MigrAIne generates a migration script:

```python
# Generated by MigrAIne: migrate_firestore_to_dynamodb.py
import asyncio
from google.cloud import firestore
import boto3

async def migrate():
    fs  = firestore.AsyncClient(project="YOUR_GCP_PROJECT")
    ddb = boto3.resource("dynamodb", region_name="us-east-1")
    table = ddb.Table("cloudmart-products")

    async for doc in fs.collection("products").stream():
        data = doc.to_dict()
        data["id"] = doc.id
        # Convert Firestore types to DynamoDB-compatible types
        table.put_item(Item=data)
        print(f"Migrated: {doc.id}")

asyncio.run(migrate())
```

### 3c. Cloud Storage → S3

```bash
# Install gsutil and AWS CLI, then:
gsutil -m cp -r gs://cloudmart-images-XXXX/ - | \
  aws s3 cp - s3://cloudmart-images/ --recursive

# Or using rclone (recommended for large buckets)
rclone sync \
  :gcs:cloudmart-images-XXXX \
  :s3:cloudmart-images \
  --progress \
  --transfers=32
```

### 3d. Pub/Sub → SNS + SQS

No data migration needed — Pub/Sub messages are ephemeral.
Re-deploy publishers (order-service) and consumers (order-notifier Lambda) to target SNS/SQS.

### 3e. Memorystore → ElastiCache

```bash
# Export Redis snapshot from Memorystore
# (Requires setting up a Memorystore export to GCS first)
gcloud redis instances export \
  cloudmart-redis \
  --region=us-central1 \
  gs://cloudmart-assets-XXXX/redis-export.rdb

# Import to ElastiCache
# Note: ElastiCache import requires the .rdb file in S3
aws s3 cp gs://cloudmart-assets-XXXX/redis-export.rdb s3://cloudmart-assets/redis-export.rdb
# Then restore via ElastiCache snapshot feature
```

## Phase 4: Application Deployment

### Push images to ECR

```bash
# Login to ECR
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  ACCOUNT_ID.dkr.ecr.us-east-1.amazonaws.com

# Tag and push (images already built from GCP)
for svc in api-gateway product-service order-service user-service; do
  docker tag cloudmart/$svc:latest \
    ACCOUNT_ID.dkr.ecr.us-east-1.amazonaws.com/cloudmart/$svc:latest
  docker push \
    ACCOUNT_ID.dkr.ecr.us-east-1.amazonaws.com/cloudmart/$svc:latest
done
```

### Deploy to EKS

MigrAIne converts the GKE manifests to EKS-compatible format:

```bash
# Apply AWS-adapted Kubernetes manifests (generated by MigrAIne)
kubectl apply -f aws-k8s-manifests/
kubectl rollout status deployment/product-service
kubectl rollout status deployment/order-service
```

Key changes in AWS manifests:
- `serviceAccountName` → AWS IAM Roles for Service Accounts (IRSA)
- Cloud SQL proxy sidecar → RDS endpoint via service endpoint
- GKE-specific annotations removed

### Deploy Lambda Functions

```bash
# MigrAIne adapts Cloud Functions to Lambda handlers
aws lambda update-function-code \
  --function-name cloudmart-image-processor \
  --zip-file fileb://functions/image-processor/lambda.zip

aws lambda update-function-code \
  --function-name cloudmart-order-notifier \
  --zip-file fileb://functions/order-notifier/lambda.zip

aws lambda update-function-code \
  --function-name cloudmart-analytics-ingester \
  --zip-file fileb://functions/analytics-ingester/lambda.zip
```

## Phase 5: Traffic Cutover

### Pre-cutover checklist

- [ ] All GKE pods healthy on EKS
- [ ] DMS replication lag < 1 minute
- [ ] All Lambda functions tested
- [ ] ElastiCache populated
- [ ] S3 bucket sync verified
- [ ] DynamoDB product count matches Firestore
- [ ] ALB health checks passing
- [ ] CloudFront distribution configured

### Cutover procedure (15-min maintenance window)

```bash
# 1. Put app in maintenance mode (update ConfigMap)
kubectl set env deployment/api-gateway MAINTENANCE_MODE=true -n cloudmart

# 2. Wait for DMS to reach 0-lag
aws dms describe-replication-tasks \
  --filters Name=replication-task-id,Values=cloudmart-migration

# 3. Stop DMS CDC
aws dms stop-replication-task \
  --replication-task-arn <TASK_ARN>

# 4. Update Route 53 to point to AWS ALB
aws route53 change-resource-record-sets \
  --hosted-zone-id <ZONE_ID> \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "cloudmart.demo",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "<ALB_ZONE_ID>",
          "DNSName": "<ALB_DNS>",
          "EvaluateTargetHealth": true
        }
      }
    }]
  }'

# 5. Remove maintenance mode
kubectl set env deployment/api-gateway MAINTENANCE_MODE=false \
  -n cloudmart --context=aws-eks-context

# 6. Monitor error rates for 30 minutes
watch -n 30 "aws cloudwatch get-metric-statistics \
  --namespace AWS/ApplicationELB \
  --metric-name HTTPCode_Target_5XX_Count \
  --dimensions Name=LoadBalancer,Value=<ALB_NAME> \
  --start-time $(date -u -d '5 minutes ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 60 \
  --statistics Sum"
```

## Phase 6: Validation & Cleanup

### Validation checklist

```bash
# API health
curl https://cloudmart.demo/health

# Create test order
curl -X POST https://cloudmart.demo/api/v1/orders \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"items":[...],"shipping_address":{...}}'

# Verify Pub/Sub → SNS/SQS event flow
# Verify Lambda (order notifier) fires
# Verify DynamoDB product reads
# Verify S3 image serving
# Check CloudWatch dashboards
```

### GCP Teardown (after 7-day validation period)

```bash
cd cloudmart-gcp/terraform
terraform destroy \
  -var="project_id=YOUR_GCP_PROJECT" \
  -var="region=us-central1"
```

> **IMPORTANT**: Do NOT destroy GCP resources until AWS is fully validated and the DNS TTL (5 min) has expired everywhere.

## Rollback Plan

If issues arise during or after cutover:

```bash
# Revert Route 53 to GCP
aws route53 change-resource-record-sets \
  --hosted-zone-id <ZONE_ID> \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "cloudmart.demo",
        "Type": "A",
        "TTL": 60,
        "ResourceRecords": [{"Value": "<GCP_IP>"}]
      }
    }]
  }'

# GCP infrastructure is kept warm for 72 hours post-cutover
```
